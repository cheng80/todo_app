# 1인 개발 작업 워크플로우와 Codex 지침서

## 1. 기본 원칙

```text
작업 하나 = Codex task 하나 = Worktree 하나 = feature 브랜치 하나 = Pull Request 하나
```

- `main`은 안정된 기준 브랜치로 유지한다.
- 기본 checkout은 `main` 확인·병합·정리에만 사용한다.
- 모든 구현은 독립 Worktree의 feature 브랜치에서 진행한다.
- `main`에 직접 작업하거나 직접 push하지 않는다.
- 기능 하나를 시작할 때마다 Worktree와 feature 브랜치를 함께 만든다.
- 커밋, Pull Request 생성, Merge, push 같은 원격 변경은 명시적으로 요청할 때만 한다.

## 2. 기본 문서 세 개로 시작하고, 필요하면 확장한다

새 프로젝트에는 아래 구조만 만든다.

```text
프로젝트폴더명/
├── reference/
│   ├── 1인-개발-작업-워크플로우와-Codex-지침서.md
│   ├── 1인-개발-플러터-Todo-앱-하네스-시나리오.md
│   └── 간소화-프로젝트-기획-템플릿/
├── START_HERE.md
├── AGENTS.md
└── docs/
    ├── product/
    │   └── 기획서.md
    └── plans/
        └── 기능명.md
```

`reference/`는 읽기 전용 참고 원본이다. Codex는 이 폴더의 지침서와 템플릿을 읽고, 실제 결과물은 루트 `AGENTS.md`와 `docs/`에 작성한다. 기능 작업 중 `reference/` 파일을 수정하지 않는다.

| 문서 | 역할 | 작성 시점 |
| --- | --- | --- |
| `AGENTS.md` | 반복 규칙, 실제 명령어, 완료 기준 | 프로젝트 시작 |
| `START_HERE.md` | 새 세션의 읽기 순서와 시작 규칙 | 프로젝트 시작 |
| `docs/product/기획서.md` | 전체 목표, MVP, 요구사항, 기술 결정 | 프로젝트 시작 후 한 번 |
| `docs/plans/기능명.md` | 이번 기능의 범위, 구현 순서, 검증 | 기능 시작마다 |

이 세 문서는 시작점이지 최대 문서 수가 아니다. 요구사항이 복잡해지면 요구사항·수용 기준 문서, 기술 선택이 많아지면 기술 설계·결정 문서, 출시가 필요해지면 QA·릴리스 문서를 추가한다. 게임 프로젝트에는 `docs/product/게임-기획-확장.md`를 추가한다.

문서는 실제로 관리할 정보가 생길 때만 추가한다. 빈 템플릿을 미리 늘리지는 않는다.

템플릿 원본과 사용법은 다음 문서를 사용한다.

- [최소 기획 템플릿 README](./간소화-프로젝트-기획-템플릿/README.md)
- [AGENTS.md 템플릿](./간소화-프로젝트-기획-템플릿/templates/AGENTS.md)
- [전체 기획서 템플릿](./간소화-프로젝트-기획-템플릿/templates/docs/product/기획서.md)
- [작업 Plan 템플릿](./간소화-프로젝트-기획-템플릿/templates/docs/plans/작업-Plan.md)

## 3. 전체 작업 흐름

```text
현재 코드·문서·변경사항 확인
→ 새 프로젝트면 기본 앱 초기화와 실행 확인
→ AGENTS.md의 실제 명령어 확인
→ 전체 기획서 1회 작성·승인
→ Worktree와 feature 브랜치 생성
→ 기능별 Plan 작성·승인
→ 구현
→ 테스트·수동 검증·리뷰
→ 한국어 커밋
→ 한국어 Pull Request
→ CI와 diff 확인
→ Merge와 브랜치 정리
```

### 새 프로젝트 시작

1. 기본 프로젝트를 초기화하고 실행·기본 테스트를 확인한다.
2. `reference/간소화-프로젝트-기획-템플릿/templates/`의 `AGENTS.md`, `START_HERE.md`, `docs/`를 프로젝트 루트에 복사한다.
3. Codex가 실제로 확인한 프로젝트 목적, 주요 디렉터리, 실행·검증 명령만 `AGENTS.md`에 채운다.
4. `docs/product/기획서.md`에 전체 MVP와 제외 범위를 작성한다.
5. 기준 커밋 이후 첫 기능부터 독립 Worktree와 feature 브랜치에서 작업한다.

사람의 요청 예시:

```text
reference/의 워크플로우 지침서와 간소화-프로젝트-기획-템플릿을 먼저 읽어줘. reference는 수정하지 말고, 실제 프로젝트 정보로 루트 AGENTS.md와 docs/product/기획서.md의 MVP·완료 조건 초안만 작성해줘. 코드 변경은 하지 마.
```

### 기능 하나 시작

1. 현재 브랜치·변경사항·기존 작업을 확인한다.
2. 새 Worktree와 feature 브랜치를 함께 만든다.
3. `reference/간소화-프로젝트-기획-템플릿/templates/docs/plans/작업-Plan.md`를 기준으로 이번 기능의 범위와 검증을 `docs/plans/기능명.md`에 작성한다.
4. Plan을 승인한 뒤 구현한다.
5. 검증과 리뷰를 마친 뒤 요청이 있을 때만 커밋·PR을 만든다.

사람의 요청 예시:

```text
루트 AGENTS.md와 docs/product/기획서.md, reference/간소화-프로젝트-기획-템플릿/templates/docs/plans/작업-Plan.md를 읽어줘. reference는 수정하지 말고, [기능 이름]의 범위와 완료 조건을 docs/plans/[기능 이름].md에만 작성해줘. 코드 변경은 하지 마.
```

```text
Plan을 승인한다. [기능 이름]을 구현하고 테스트·검증·리뷰 결과를 보고해줘. 커밋, PR, 원격 변경은 하지 마.
```

### 새 세션 재개

새 세션에서는 별도 `HANDOFF.md`를 만들지 않는다. 현재 기능 Plan의 `작업 결과와 세션 인수인계`를 최신 상태로 갱신한다.

```text
START_HERE.md와 현재 기능 Plan을 읽고, 진행 상태와 다음 작업만 보고해줘. 코드 수정은 하지 마.
```

## 4. Worktree와 브랜치

### 브랜치

- 기능: `feature/<짧은-이름>`
- 버그 수정: `fix/<짧은-이름>`
- 문서·설정: `chore/<짧은-이름>`
- 긴급 수정: `hotfix/<짧은-이름>`

Codex 앱의 브랜치 접두사를 `codex/`로 설정했다면 실제 이름은 `codex/feature/<짧은-이름>`처럼 생성된다.

작업 전에는 최소한 다음을 확인한다.

```bash
git branch --show-current
git status --short
git worktree list
```

원격 저장소에 연결된 작업에서만 최신 기준 브랜치, 같은 이름의 원격 브랜치, 열린 Pull Request도 확인한다.

### Worktree는 모든 기능 작업에 사용한다

- 기준 `main`에서 Worktree와 feature 브랜치를 함께 만든다.
- 기본 checkout은 기준 브랜치 확인, 병합, 정리에만 사용한다.
- Worktree 하나에는 feature 브랜치 하나만 연결한다.
- 같은 브랜치를 두 Worktree에서 동시에 열지 않는다.
- Merge 후 해당 Worktree와 feature 브랜치를 정리한다.

## 5. 검증, 리뷰, Pull Request

완료 전에 실행한다.

- 관련 테스트
- lint
- typecheck
- build
- 핵심 사용자 흐름 수동 검증
- 변경 diff 리뷰

실행하지 못한 검증은 통과했다고 쓰지 않는다.

Pull Request는 기능 하나만 포함한다. 제목·본문은 한국어로 작성하고 아래 네 항목을 포함한다.

```text
## 변경 내용
## 변경 이유
## 검증
## 제외 범위 및 주의사항
```

1인 개발에서는 본인이 CI와 diff를 확인한 뒤 Merge할 수 있다. Draft PR은 미완성·실험 작업에만 사용하고, 기본 병합은 스쿼시 Merge로 둔다.

## 6. 개인 맞춤 지침

개인 맞춤 지침에는 아래 정도만 둔다.

```text
항상 한국어로 간결하게 답변한다. 결과를 먼저 말하고 불필요한 설명은 줄인다.

작업 전 기존 코드와 구조를 먼저 확인한다. 기존 패턴과 도구를 우선 재사용하고, 가장 단순하고 작은 변경을 선택한다.

기능 작업은 다음 순서를 따른다.
기획서 확인 → Plan 작성 → 범위 확정 → 구현 → 테스트·검증 → 코드 리뷰 → 커밋 → Pull Request

기능 하나를 하나의 작업 단위, Worktree, feature 브랜치, Pull Request로 관리한다. 기본 checkout은 main 확인·병합·정리에만 사용하고 구현은 Worktree에서 진행한다. main 브랜치에 직접 작업하지 않는다.

사용자 변경사항을 덮어쓰거나 되돌리지 않는다. 범위를 벗어난 변경, 원격 저장소 변경, Pull Request 생성, Merge는 명시적인 요청 없이 실행하지 않는다.

완료를 주장하기 전에 관련 테스트, lint, typecheck, build와 필요한 수동 검증을 실행한다. 검증하지 못한 항목은 명확히 밝힌다.

모든 커밋 메시지와 Pull Request 제목·본문은 한국어로 작성한다. 코드 식별자, 파일 경로, 명령어처럼 번역하면 안 되는 항목만 원문을 유지한다.
```

프로젝트별 명령어와 예외 규칙은 개인 지침이 아니라 루트 `AGENTS.md`에 둔다.

## 7. 커밋 지침

```text
커밋은 하나의 목적을 가진 논리적인 변경 단위로 작성한다.
커밋 전 staged diff를 확인하고 unrelated 변경사항과 비밀정보를 포함하지 않는다.
검증 결과를 확인한 뒤 커밋한다.

커밋 메시지는 한국어로 작성한다.
형식: <분류>: <변경 내용>
분류: 기능, 수정, 문서, 테스트, 리팩터링, 설정, 제거

예시:
기능: Todo 추가와 수정 기능 구현
수정: 빈 제목 입력 오류 처리
문서: Todo 앱 MVP 범위 정리
```

## 8. Pull Request 지침

```text
Pull Request는 기능 하나 또는 수정 하나만 포함한다. 관련 없는 변경사항을 섞지 않는다.

제목과 본문은 한국어로 작성한다.
제목 형식: <분류>: <변경 내용>

본문 형식:
## 변경 내용
## 변경 이유
## 검증
## 제외 범위 및 주의사항

검증에는 실제 실행한 테스트, lint, typecheck, build, 수동 검증 결과만 적는다.
실행하지 못한 항목은 통과했다고 쓰지 않고, 이유와 남은 위험을 적는다.

검증과 diff 리뷰가 끝난 뒤에만 Pull Request를 생성한다.
Draft Pull Request는 미완성·실험 작업에만 사용한다.
CI와 diff를 확인한 뒤 스쿼시 Merge하고, Worktree와 feature 브랜치를 정리한다.
```

## 9. Linear 도입 여부

1인 개발이고 GitHub Pull Request와 `docs/plans/`만으로 우선순위를 관리할 수 있다면 Linear는 아직 필요 없다.

아이디어·기능·프로젝트가 늘어나 현재와 다음 작업을 고르기 어려워질 때 도입한다. 처음에는 백로그, 진행 중, 완료 세 상태만 사용한다.
