# 프로젝트 기획서

## 1. 프로젝트 개요

- 프로젝트명: Todo 앱
- 상태: 초안
- 한 줄 목표: iOS에서 Todo를 로컬에 저장하고 간단히 관리한다.
- 대상 플랫폼: iOS 모바일
- 핵심 사용자: 개인 할 일을 빠르게 기록하고 관리하려는 사용자
- 해결하려는 문제: 앱을 다시 열어도 남아 있는 개인 Todo를 추가·완료·수정·삭제할 수 있어야 한다.

## 2. MVP와 제외 범위

### 이번에 만드는 것

- 제목은 필수, 본문은 선택인 Todo 생성과 수정
- SQLite에 저장된 Todo 목록 조회와 완료 여부 전환
- 스와이프 삭제, 빈 목록과 저장 실패 안내

### 이번에 만들지 않는 것

- 검색, 카테고리, 마감일, 우선순위, 알림
- 로그인, 동기화, 공유, Riverpod 등 별도 상태관리 패키지

## 3. 핵심 사용자 흐름과 상태

| 흐름 | 사용자의 행동 | 정상 결과 | 빈 상태·오류·권한 등 필요한 예외 |
| --- | --- | --- | --- |
| FLOW-01 | Todo를 추가한다. | 제목과 선택 본문이 SQLite에 저장되고 목록에 표시된다. | 제목이 비어 있으면 저장하지 않고 입력 오류를 표시한다. |
| FLOW-02 | 목록의 체크박스를 누른다. | 완료 여부가 바뀌고 완료 상태가 구분된다. | Todo가 없으면 빈 목록 안내를 표시한다. |
| FLOW-03 | Todo를 열어 수정하거나 왼쪽으로 밀어 삭제한다. | 수정 내용이 저장되거나 Todo가 목록과 SQLite에서 제거된다. | 저장·조회 실패 시 간단한 오류 메시지를 표시한다. |
| FLOW-04 | 앱을 다시 연다. | 이전에 저장한 Todo 목록이 다시 표시된다. | 저장된 Todo가 없으면 빈 목록 안내를 표시한다. |

## 4. Must 요구사항과 수용 기준

| ID | 기능 | 사용자의 행동 | 완료로 판단하는 결과 | 검증 방법 |
| --- | --- | --- | --- | --- |
| REQ-001 | Todo 생성 | 제목과 선택 본문을 입력해 저장한다. | 저장된 Todo가 목록에 표시되고 앱 재실행 뒤에도 남는다. | 수동, 단위 |
| REQ-002 | Todo 조회 | 앱을 열거나 저장 뒤 목록을 본다. | SQLite에 저장된 모든 Todo가 목록에 표시된다. | 수동, 단위 |
| REQ-003 | 완료 전환 | 목록의 체크박스를 누른다. | 완료 여부와 화면 표시가 바뀌고 다시 열어도 유지된다. | 수동, 단위 |
| REQ-004 | Todo 수정 | 목록 항목을 열어 제목 또는 본문을 수정한다. | 수정 내용이 목록과 SQLite에 반영된다. | 수동, 단위 |
| REQ-005 | Todo 삭제 | 목록 항목을 왼쪽으로 민다. | Todo가 목록과 SQLite에서 제거된다. | 수동, 단위 |
| REQ-006 | 입력·빈·오류 상태 | 제목 없이 저장하거나 빈 목록·저장 실패 상태를 본다. | 제목 저장은 차단되고 각 상태를 이해할 수 있는 안내가 표시된다. | 수동, 단위 |

## 5. 데이터와 기술 결정

### 데이터

| 항목 | 필드 또는 내용 | 저장 위치 | 삭제·변경 규칙 |
| --- | --- | --- | --- |
| Todo | `id`, `title`, `content`, `isCompleted` | 기기 로컬 SQLite `todos` 테이블 | 제목은 비어 있을 수 없고, 삭제하면 즉시 제거한다. |

### 기술 결정

| 결정 | 선택 | 이유 | 나중에 다시 검토할 조건 |
| --- | --- | --- | --- |
| 앱 프레임워크 | Flutter와 Dart | 현재 기준 앱과 iOS 시뮬레이터 검증 환경이 준비돼 있다. | 지원 플랫폼이 추가될 때 |
| 저장소 | SQLite와 `sqflite` | Todo CRUD와 재실행 뒤 데이터 유지에 충분하다. | 동기화나 다중 기기 데이터가 필요할 때 |
| 데이터 접근 | `TodoRepository`가 SQLite CRUD를 담당 | 화면과 저장 로직을 작게 분리한다. | 화면 상태와 비동기 흐름이 복잡해질 때 |
| 상태관리 | 화면이 `TodoRepository` 호출 뒤 목록을 새로 조회 | 현재 CRUD 범위에 별도 패키지가 필요 없다. | 화면·상태 공유가 늘어날 때 Riverpod을 검토한다. |

## 6. 검증 기준

- [ ] iOS 시뮬레이터에서 추가 → 완료 처리 → 제목 수정 → 앱 재실행 → 삭제를 확인했다.
- [ ] REQ-001부터 REQ-006까지의 완료 조건을 확인했다.
- [ ] 관련 단위 테스트를 실행했다.
- [ ] `flutter analyze`와 `flutter build ios --simulator`를 실행했다.
- [ ] 빈 목록, 빈 제목, 저장 실패 상태를 확인했다.
